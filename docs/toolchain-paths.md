## Phase C: real clone-stamp algorithm -- COMPLETE

Ported the Python plugin's validated algorithm (`clonestamp_core.py`'s
`stamp_dab`/radial mask/alpha-scale logic) directly into
`KisToolCloneStamp.cpp`, using `KisPaintDeviceSP::readBytes`/`writeBytes`
(not `KisRandomAccessorSP` -- byte-block read/write matched the Python
`pixelData()`/`setPixelData()` shape more directly and was simpler to get
right) wrapped in a single `KisTransaction` per stroke, committed in
`endPrimaryAction`/`deactivate()`.

- Ctrl+click samples a source point (`sampleSource`).
- Plain drag paints: `beginStroke` fixes the source-to-destination offset
  (respecting Aligned semantics), `stampDabAt` reads a `size`x`size` square
  from source, applies a radial-gradient hardness mask + opacity scale via
  `QImage` (Krita's 8-bit RGBA device bytes match `QImage::Format_ARGB32`
  byte-for-byte on little-endian, same trick the Python version used), then
  composites and writes back. Both rects are jointly clipped to canvas
  bounds so off-canvas dabs don't desync source/destination alignment.
- Shift+drag resizes the brush (`m_isResizing`, tracks widget-space dx).
- `paint()` draws a live hover-circle outline; no floating preview window
  needed since this runs inside Krita's own paint-event, unlike the
  Python version's workaround.

Gap found and fixed this session: the initial port hardcoded brush size
(50), hardness (0.5), opacity (100), and aligned (always true) with
`createOptionWidget()` returning `nullptr` -- unlike the Python docker,
which exposes all four as live controls. Added a real option widget
(Hardness/Opacity spinboxes + Aligned checkbox, wired via lambdas to the
tool's private members) so these match the Python version's tunability.
Brush *size* is still resize-drag-only (matches the Python version, which
also has no size spinbox wired to the brush-drag path... actually it does
via `sizeSpin` -- a size spinbox in the option widget is a further gap,
left for Phase D).

Rebuilt (`cmake --build . --target kritatoolclonestamp`), reinstalled,

## Phase C follow-up: real bugs found via first live test pass

First hands-on test (2026-07-15) surfaced three real problems, all now fixed:

1. **Toolbox priority collision.** `setPriority(3)` collided with
   `tool_lazybrush`'s Colorize Mask Tool (also 3) in the same `Fill`
   section -- `KoToolBoxLayout_p.h`'s `addButton()` only `qWarning()`s on
   this, it doesn't relocate either button, so with the *also*-borrowed
   Smart Patch icon (see next point) it was very easy to mistake for the
   tool not existing at all. Changed to `setPriority(5)` (free slot between
   Smart Patch=4 and Fill=14).
2. **Icon borrowed from Smart Patch** made the two literally
   indistinguishable in the toolbox (same glyph, adjacent slots). Changed
   to `edit-copy` (a standard theme icon, safe fallback, no new SVG/qrc
   work needed) -- a real dedicated stamp icon is still a Phase D item.
3. **Root cause of "nothing works" -- wrong event-handling API.** Ctrl+click
   and Shift+drag were implemented by checking `event->modifiers()` inside
   `beginPrimaryAction`/`continuePrimaryAction`. This never fires: Krita's
   `KisInputManager` intercepts both gestures itself as built-in "alternate
   actions" (`kis_change_primary_setting_action.cpp` confirms Shift+Left-click
   is hardwired to `ChangeSize`; `KisTool::AlternateAction::SampleFgImage`
   is documented `// Default: Ctrl+Left click`) and routes them to
   `beginAlternateAction(event, AlternateAction)` instead -- a completely
   different virtual method `KisTool`'s base implementation no-ops by
   default. Since `KisToolCloneStamp` never overrode
   `begin/continue/endAlternateAction`, Ctrl+click sampling and Shift-drag
   resize were silently swallowed no matter what the primary-action code
   said. Fixed by moving both gestures to real
   `beginAlternateAction`/`continueAlternateAction`/`endAlternateAction`
   overrides (handling `SampleFgImage` and `ChangeSize`), and stripped the
   dead modifier checks out of the primary-action methods. Also added a
   `Size` spinbox to the option widget (previously only reachable via the,
   until now broken, Shift-drag).

Lesson for next time: KisTool subclasses must not assume modifier keys
reach `beginPrimaryAction` -- check `kis_tool.h`'s `AlternateAction` enum
comments for the *actual* default gesture bindings before wiring up custom
modifier-based interactions.

a real hands-on test pass (this session's automated screenshot/mouse

user's own live testing session -- don't do that again; ask the user to

## Phase D follow-up: Shift+drag now syncs the option widget and also
